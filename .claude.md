# FreeStyle Libre 3 Integration - AI Context

## Project Overview

This document provides context for AI assistants working on integrating FreeStyle Libre 3 support into xDrip4iOS (xdripswift). The implementation is based on the working Android Juggluco app integration.

## Source Reference

**Primary Reference:** [Juggluco Android App](https://github.com/lutzlukesch/Juggluco)
- Language: Java + C++ (JNI)
- Key Files:
  - `Common/src/libre3/java/tk/glucodata/Libre3GattCallback.java` - BLE GATT communication
  - `Common/src/main/cpp/net/libreview/newlibre3.cpp` - LibreView cloud upload
  - `Common/src/main/java/tk/glucodata/ScanNfcV.java` - NFC sensor scanning
  - `Common/src/main/cpp/sensoren.hpp` - Sensor data management

## Technical Architecture

### Platform Differences

| Aspect | Android (Juggluco) | iOS (xDrip4iOS) |
|--------|-------------------|-----------------|
| **NFC** | `android.nfc.Tag` API | CoreNFC framework |
| **Bluetooth** | BluetoothGatt API | CoreBluetooth framework |
| **Cryptography** | Native C++ (JNI) | CryptoKit (Swift) |
| **Language** | Java + C++ | Swift |
| **Background Processing** | Services + AlarmManager | Background modes (limited) |

### Key Components to Implement

1. **NFC Scanning** (CoreNFC)
   - Initial sensor activation
   - Extract sensor UID and metadata
   - Detect Libre 3: `uid.length == 8 && uid[6] != 7`

2. **Bluetooth GATT Communication** (CoreBluetooth)
   - Service UUID: `FDE3` (data service)
   - Security service for ECDH key exchange
   - Multiple characteristics for glucose, historic, patch status data

3. **Cryptographic Layer** (CryptoKit)
   - ECDH key exchange (P-256)
   - AES-GCM encryption/decryption
   - Challenge-response authentication

4. **Data Processing**
   - Parse encrypted glucose data
   - Handle historic backfill
   - Store in CoreData

5. **LibreView Cloud Upload**
   - JSON payload construction
   - OAuth token management
   - Background sync

### Implementation Strategy

**Phase 1: NFC Support**
- Extend `LibreNFC.swift` to detect Libre 3 sensors
- Create `Libre3NFCManager.swift` for Libre 3-specific NFC logic
- Extract sensor serial number and metadata

**Phase 2: Cryptography Foundation**
- Create `Libre3CryptoContext.swift` using CryptoKit
- Implement ECDH key exchange
- Implement AES-GCM encryption/decryption wrappers

**Phase 3: BLE Communication**
- Create `CGMLibre3Transmitter.swift` based on `CGMLibre2Transmitter.swift`
- Implement GATT service discovery
- Handle notification cascade (order-dependent)
- Process encrypted data packets

**Phase 4: Data Processing**
- Parse glucose readings from decrypted payloads
- Store in CoreData
- Integrate with existing glucose display

**Phase 5: LibreView Integration**
- Create `Libre3LibreViewUploader.swift`
- Implement JSON payload builder
- Handle background upload

**Phase 6: Testing & Refinement**
- Unit tests for cryptographic functions
- Integration tests with sensor
- Background mode testing

## Critical Implementation Notes

### Cryptography Details

From Juggluco's implementation:

1. **ECDH Setup** (`Libre3GattCallback.java` line 602-613)
   - Uses exported kAuth if available (returning user)
   - Generates new ECDH keypair for first-time users
   - Stores kAuth for future sessions

2. **Challenge-Response** (lines 375-404)
   - Receives r1 (16 bytes) + nonce1 (7 bytes)
   - Generates r2 (16 bytes)
   - Combines with sensor PIN (4 bytes)
   - Encrypts and returns

3. **Session Keys** (lines 394-402)
   - Extracts kEnc (encryption key)
   - Extracts ivEnc (initialization vector)
   - Initializes AES-GCM context

### GATT Characteristic Cascade

**Critical Order** (from `handleonDescriptorWrite`, line 718-783):

```
1. PATCH_CONTROL → enable notifications
2. EVENT_LOG → enable notifications  
3. HISTORIC_DATA → enable notifications
4. CLINICAL_DATA → enable notifications
5. FACTORY_DATA → enable notifications
6. GLUCOSE_DATA → enable notifications
7. PATCH_STATUS → enable notifications
8. COMMAND_RESPONSE (security) → enable notifications
9. CERT_DATA → enable notifications
10. CHALLENGE_DATA → enable notifications
    → triggers handleMSLibre3SecurityNotificationsEnabledEvent
```

## Code Style Guidelines

- Follow Swift naming conventions (camelCase)
- Use `// MARK: -` sections for organization
- Add OSLog traces for debugging
- Use weak references to avoid retain cycles
- Implement proper error handling with Result types

## Testing Considerations

1. **NFC Testing**: Requires physical Libre 3 sensor
2. **BLE Testing**: Can use iOS Simulator for structure, requires real sensor for data
3. **Crypto Testing**: Can unit test with known test vectors
4. **Background Testing**: Must test on physical device

## Security Considerations

1. Store kAuth securely in Keychain
2. Never log encryption keys or sensor PIN
3. Validate all sensor data before processing
4. Implement certificate pinning for LibreView uploads

## Background Processing Limitations (iOS)

- Background BLE limited to 10 seconds per connection
- Use `CBCentralManager` background mode
- Expect more frequent disconnections than Android
- Plan for data queuing and retry logic

## Dependencies

- **CoreNFC**: Built-in (iOS 13+)
- **CoreBluetooth**: Built-in
- **CryptoKit**: Built-in (iOS 13+)
- No external dependencies needed for core functionality

## Known Issues & Limitations

1. **iOS Background Restrictions**: More aggressive than Android
2. **NFC Session Timeout**: 60 seconds max
3. **LibreView Account Required**: User must have Abbott account ID
4. **Sensor Activation**: Requires NFC scan during warmup period
